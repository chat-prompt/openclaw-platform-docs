> **이 편에서 다루는 것**: 봇이 사는 워크스페이스 구조, 반드시 알아야 할 핵심 파일 5개, 세션과 기억의 관계.

봇을 키우려면, 먼저 봇이 **어디에 사는지**를 알아야 해.

사람이 집이 있어야 물건을 두고, 메모를 붙이고, 생활 습관이 생기듯 — 봇도 "집"이 있어야 성격이 생기고, 기억이 쌓이고, 규칙을 따르게 돼.

그 집이 바로 **워크스페이스**야.

---

## 워크스페이스 = 봇이 사는 집

```
~/.openclaw/workspace-[에이전트이름]/
```

이 폴더가 봇의 전부야. 봇의 성격, 기억, 규칙, 기록 — 다 여기 들어 있어. 이 집을 어떻게 꾸미느냐에 따라 봇이 어떤 봇이 되는지가 결정돼.

**비유하면**: 새로 입양한 고양이에게 집을 꾸며주는 것과 같아. 밥그릇(도구), 캣타워(스킬), 이름표(정체성), 훈련 규칙(AGENTS.md) — 하나씩 세팅하는 거지.

---

## 핵심 파일 5개

이 5개 파일만 이해하면 된다:

| 파일 | 비유 | 역할 | 언제 쓰나 |
|------|------|------|----------|
| **SOUL.md** | 영혼 | 성격, 말투, 가치관, 미션 | 봇이 매 세션마다 읽음 |
| **USER.md** | 집사 프로필 | 주인(너) 정보 — 이름, 역할, 선호 | 봇이 너를 파악할 때 |
| **AGENTS.md** | 사규/매뉴얼 | 행동 규칙, 보안, 절차, 도구 사용법 | 봇이 "이러면 어떻게 하지?" 할 때 |
| **MEMORY.md** | 화이트보드 | 현재 상태 스냅샷 — 진행 중인 일, 기억 | 세션마다 읽어서 맥락 파악 |
| **IDENTITY.md** | 명함 | 이름, 프로필 사진, 기본 정보 | 자기소개할 때 |

<div class="tip-box">
<span class="tip-icon">💡</span>
<span><strong>핵심 원칙</strong>: 봇은 매 세션 시작 시 이 파일들을 읽어. 파일에 써있는 게 곧 봇의 기억이고 성격이고 규칙이야. <strong>"파일에 안 쓰면 기억 못 한다."</strong></span>
</div>

### SOUL.md — 영혼

봇의 말투, 성격, 가치관, 미션을 정의하는 문서. "**왜 이 봇이 존재하는가**"의 답이 여기 들어 있어.

```markdown
# SOUL.md

## 미션
> 반복을 도구로 바꿔서, 미션이 있는 사람들이 미션에만 집중하게 한다.

## 성격
- 다정하고 장난기 있지만 독립적
- 의견이 있어 — "좋은 질문이에요!" 같은 빈 말 안 해
- 먼저 찾아보고, 그래도 모르면 물어봐

## 해야 하는 것
- 반복 업무 발견하면 먼저 자동화 제안
- 모르면 검색해서 답을 가지고 오기

## 하면 안 되는 것
- 확신 없는 것을 외부에 발송
- 주인 개인정보 외부 유출
```

미션 한 줄을 적어두면, 봇이 애매한 상황에서 "이걸 해야 할까?"를 판단하는 기준이 돼. AI는 자연어로 작동하니까, 자연어로 된 미션이 곧 행동 기준이야.

<div class="tip-box">
<span class="tip-icon">🔥</span>
<span><strong>실전 사례</strong>: 뽀야의 SOUL.md에 "의견이 있어야 해"라고 적어둔 뒤로, 집사가 "이거 어떻게 할까?"라고 물으면 "좋은 질문이에요"가 아니라 <strong>"A 방법이 나을 것 같아. 이유는..."</strong>이라고 답하게 됐어. SOUL.md 한 줄이 행동을 바꾸는 거야.</span>
</div>

### USER.md — 집사 프로필

봇이 주인에 대해 아는 정보야. 소통 스타일, 선호, 역할 등을 적어두면 봇이 맞춤형으로 대화해.

```markdown
# USER.md — 주인 정보

- **이름:** 닿 (송다혜)
- **Slack ID:** U06BNH5R26T
- **역할:** PM, 운영 자동화
- **성격:** 실용적, 결과 중심, "어떻게 하면 되는지" 선호
- **선호:** 짧고 핵심적인 답변, 코드블록 포함 설명
```

이걸 왜 적냐고? **봇이 주인의 소통 스타일에 맞춰서** 대화할 수 있어서야.

- "장황한 설명보다 핵심"을 선호하면 → 봇이 짧게 답함
- "코드블록 포함"을 선호하면 → 봇이 설정 예시를 코드블록으로 보여줌
- "반말 OK"면 → 봇이 딱딱한 존댓말 안 씀

봇과 대화하면 할수록 "이 봇이 나를 아네"하는 느낌이 달라져. 특히 팀 채널에서 주인의 질문에는 맞춤형으로, 다른 사람의 질문에는 일반적으로 대답하는 차별화가 가능해져.

### AGENTS.md — 사규/매뉴얼

행동 규칙, 보안, 절차 등 "**이런 상황에서 이렇게 해라**"가 모여 있는 문서. 회사의 사내 규정 같은 거야.

보안 규칙, 호명규칙, 대화질서, 도구 사용법 같은 것들이 여기 들어가. 다음 편(2편: 갑옷 입히기)에서 이 파일을 채우는 법을 자세히 다뤄.

```markdown
## 보안
### 주인 (풀 권한):
- Slack: `U06BNH5R26T`

### 그 외:
- 일반 대화만 허용
- 파일 접근, 명령어 실행은 거절
```

### MEMORY.md — 화이트보드

현재 상태 스냅샷. "지금 뭐가 돌아가고 있는가"를 적어두는 곳. 세션이 새로 시작될 때마다 읽어서 맥락을 파악해.

```markdown
## 현재 프로젝트
- 대시보드 v2 개발 중 (70% 완료)
- 주간 보고서 자동화 스킬 작업 중

## 이번 주 블로커
- Airtable API 속도 저하 문제 조사 필요
```

MEMORY.md를 잘 관리하면, 봇이 매번 "어디까지 했지?"부터 설명할 필요 없이 바로 본론으로 들어갈 수 있어.

### IDENTITY.md — 명함

봇의 이름, ID, 기본 정보. 팀 채널에서 "너 누구야?"라고 물었을 때 자기소개할 수 있어야 하니까.

```markdown
# IDENTITY.md
- **이름:** 뽀야
- **역할:** 자동화 전문 비서
- **주인:** 닿 (송다혜)
- **Slack Bot User ID:** U0XXXXBOT
```

봇이 자기 Slack ID를 알아야 `@멘션`이 자기한테 온 건지 판단할 수 있어.

## 세션 = 카톡 채팅방

세션 개념은 카카오톡에 비유하면 바로 이해돼:

- **DM** = 1:1 채팅방 → 봇과 나만의 대화 (하나의 메인 세션)
- **채널** = 오픈채팅방 → 여러 사람이 있는 곳
- **스레드** = 답글 묶음 → 하나의 주제로 묶인 대화 (스레드마다 별도 세션!)

![세션 개념도 — DM, 채널, 스레드](/images/guides/guide-01/session-concept.png)

### "세션은 분리, 파일은 공유"

이게 핵심 개념이야.

무슨 뜻이냐면:
- 스레드 A에서 한 대화를 스레드 B에서는 **기억 못 해** (세션이 분리돼 있으니까)
- 하지만 MEMORY.md에 기록한 건 **모든 세션에서 읽을 수 있어** (파일은 공유되니까)

![세션은 분리, 파일은 공유](/images/guides/guide-01/session-file-relation.png)

---

## "파일에 안 쓰면 기억 못 한다"

이게 LLM 에이전트의 가장 중요한 특성이야.

LLM은 세션이 끝나면 대화를 **전부 잊어**. 어제 한 대화? 1시간 전 대화? 다른 스레드 대화? 전부 모름.

**기억하는 유일한 방법 = 파일에 쓰기**

그래서 "이거 기억해"라고 말하면 봇이 memory 파일에 적어둬. 다음에 깨어날 때 그 파일을 읽으면 "아, 이런 게 있었구나" 하고 이어갈 수 있어.

<div class="tip-box">
<span class="tip-icon">🔥</span>
<span><strong>실전 교훈</strong>: 뽀짝이가 세션 리셋 후 기억을 잃어서 같은 작업을 두 번 한 적이 있어. 그 뒤로 "머릿속 메모 금지, 반드시 파일에!" 규칙이 생겼지. "기억하고 싶은 건 무조건 파일에 쓰자"는 게 생활 규칙 1번이야.</span>
</div>

### 세션 리셋과 기억

"그러면 세션을 리셋하면 기억이 다 날아가?"

**세션 대화 기록은 날아가.** 하지만 파일에 쓴 건 남아. 이게 핵심이야.

| | 세션 리셋 후 |
|---|---|
| 대화 기록 | ❌ 사라짐 |
| MEMORY.md | ✅ 유지 |
| memory/날짜.md | ✅ 유지 |
| SOUL.md | ✅ 유지 |
| learnings/ | ✅ 유지 |

그래서 중요한 건 파일에 써야 해. 대화로만 주고받은 건 세션이 바뀌면 사라져.

### 여러 세션이 동시에?

그렇다. 봇은 **동시에 여러 세션**을 처리할 수 있어:

- DM 메인 세션에서 나와 대화 중
- 채널 스레드 A에서 동료와 대화 중
- 채널 스레드 B에서 다른 동료와 대화 중

각각 독립된 세션이라, 스레드 A의 대화가 스레드 B에 영향을 주지 않아. 하지만 **워크스페이스 파일은 공유**되니까, "MEMORY.md에 이거 기록해"라고 하면 모든 세션에서 볼 수 있어.

## 워크스페이스 전체 구조

참고로, 5개 핵심 파일 외에 워크스페이스에 들어가는 폴더들을 정리하면:

```
~/.openclaw/workspace-봇이름/
├── SOUL.md              ← 영혼 (성격, 미션)
├── USER.md              ← 집사 프로필
├── AGENTS.md            ← 사규/매뉴얼 (행동 규칙)
├── MEMORY.md            ← 화이트보드 (현재 상태)
├── IDENTITY.md          ← 명함
├── memory/              ← 일기장 (일별 기록)
│   ├── 2026-03-28.md
│   └── ...
├── learnings/           ← 오답노트 (삽질 기록)
│   └── airtable.md
├── skills/              ← 워크스페이스 전용 스킬
│   └── my-custom/SKILL.md
└── ...
```

| 폴더 | 비유 | 역할 |
|------|------|------|
| `memory/` | 일기장 | 매일의 날것 기록 |
| `learnings/` | 오답노트 | 삽질 → 해결 과정 |
| `skills/` | 매뉴얼 서랍 | 봇이 할 수 있는 특수 작업 |

---

## 이해 확인 체크리스트

- [ ] "워크스페이스가 뭐야?" → 봇이 사는 집, `~/.openclaw/workspace-이름/`
- [ ] "핵심 파일 5개 뭐야?" → SOUL, USER, AGENTS, MEMORY, IDENTITY
- [ ] "세션이 뭐야?" → 대화 단위. 스레드마다 분리되지만, 파일은 공유
- [ ] "봇이 기억하려면?" → 파일에 써야 해!

이 개념을 잡아야 다음 편의 보안·호명·대화질서가 이해돼. 집 구조를 모르면 인테리어를 할 수 없으니까.
