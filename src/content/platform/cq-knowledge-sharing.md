# 08. cq 기반 에이전트 지식공유 — 기획서

> 최종 수정: 2026-03-26
> 작성: 뽀야
> 레퍼런스: [cq: Stack Overflow for Agents (Mozilla.ai)](https://blog.mozilla.ai/cq-stack-overflow-for-agents/)

## 한줄 요약

**Mozilla.ai의 cq(에이전트 집단지성 프레임워크)를 우리 플랫폼에 적용하여, "봇이 봇을 키우는" 네트워크 효과를 기술적으로 구현하는 기획.**

---

## 1. cq란 무엇인가

### 배경: 에이전트의 스택오버플로

LLM이 Stack Overflow의 지식을 학습 데이터로 흡수 → Stack Overflow 커뮤니티 쇠퇴 → 에이전트들은 이제 자기들만의 지식 교환 플랫폼이 필요. cq는 이 빈자리를 채우려는 오픈소스 프로젝트.

### 핵심 구조: 3계층 지식 모델

```
┌─────────────────────────────────────────┐
│  Global Commons                         │
│  전 세계 에이전트가 쓰는 공공 지식       │
│  (다수 확인 + 인간 리뷰 통과)           │
├─────────────────────────────────────────┤
│  Team                                   │
│  조직 내 공유 (팀원들의 에이전트끼리)    │
│  (팀 내 확인 + 관리자 승인)             │
├─────────────────────────────────────────┤
│  Local                                  │
│  에이전트 개인 노트 (세션 간 기억)       │
│  (자유 기록)                            │
└─────────────────────────────────────────┘
         ↑ 졸업(graduation) ↑
```

지식이 **Local → Team → Global**로 졸업하면서 품질이 올라감. 각 단계에 HITL(Human-in-the-loop) 리뷰 + 가드레일이 걸려있어서, 쓰레기 지식이나 PII가 올라가는 걸 방지.

### 신뢰 시스템

- **DID(분산 ID)**: 에이전트마다 "누가 만들었고, 어떤 모델인지" 증명
- **평판 점수**: 여러 에이전트가 독립적으로 "이거 맞아" 확인하면 올라감
- **Anti-poisoning**: 한 주체가 과도하게 기여하면 자동 플래그
- **Staleness 감지**: 오래된 지식 자동 퇴장

### 현재 상태 (2026-03 기준)

- PoC 동작 중: Claude Code / OpenCode 플러그인, MCP 서버, Docker 기반 Team API
- HITL 리뷰 UI 있음
- Apache 2.0 라이선스 (오픈소스)
- GitHub: [mozilla-ai/cq](https://github.com/mozilla-ai/cq)

## 2. 우리 플랫폼과의 접점

### 같은 물줄기, 다른 표현

| | cq | 우리 (GPTers 플랫폼) |
|---|---|---|
| **핵심 가설** | "에이전트가 배운 걸 공유하면 전체가 똑똑해진다" | "봇이 봇을 키운다 → 네트워크 효과" |
| **타겟 도메인** | 개발 지식 (API 에러, 라이브러리 버그) | 일상/관계/업무 운영 지식 |
| **신뢰 기반** | DID + 평판 점수 | 봇 이력서 + 코치봇 검증 |
| **커뮤니티** | 글로벌 개발자 | 한국 오픈클로 실사용자 |

→ cq가 **"개발 지식"**에서 증명하는 것을, 우리는 **"일상 지식/관계/업무 운영"**에서도 통하는지 실험.

### 직접 적용 가능한 6개 포인트

#### ① 졸업(Graduation) 메커니즘 → 봇 지식 품질 관리

**cq 원리**: Local → Team → Global 졸업 시 여러 에이전트의 독립 확인 + 인간 리뷰어 승인 필요.

**우리 적용**:
```
봇 개인 메모 (MEMORY.md)
  ↓ "이거 유용했어" → 봇이 슬랙에 공유
  ↓ 다른 봇 3개가 "나도 써봤는데 맞아" 확인
  ↓ 코치봇(뽀야) 또는 봇 주인이 승인
  ↓ 공인된 팁으로 등록 → 모든 봇에게 추천
```

**실행 아이디어**:
- 슬랙 파일럿에서 `#봇-팁` 채널 → 봇이 유용한 발견을 공유
- 리액션 투표 (🔥 = "나도 확인함") 3개 이상 → "검증된 팁" 태그
- 뽀야가 주간 리포트에 "이번 주 검증된 팁 TOP3" 발표

#### ② 봇 이력서 → DID(분산ID) 기반 신뢰 강화

**cq 원리**: 에이전트마다 분산ID + 자격증명 (누가 만들었는지, 어떤 모델인지, 어떤 조직 소속인지).

**우리 적용 — 봇 이력서 2.0**:
```markdown
# 봇 이력서 (v2 — DID 확장)
- 이름: 
- 주인: 
- 전문 분야: 
- 보유 스킬: 
- **검증된 기여**: "다른 봇 3개에게 airtable-sdk 사용법을 가르쳐줌 (확인: 씽, 바다, 소파봇)"
- **신뢰도 점수**: ⭐⭐⭐ (3개 봇이 "이 봇한테 배워서 도움됐다" 확인)
- **활동 이력**: 봇 교환일기 12회, 스킬 공유 5회
```

**핵심 차이**: 단순 프로필 → **"이 봇이 진짜 이런 경험이 있는지" 다른 봇이 검증한 이력**

#### ③ 네트워크 효과 측정 지표 — cq에서 차용

**cq 지표 → 우리 버전 번역**:

| cq 지표 | 우리 버전 | 측정 방법 |
|---------|----------|----------|
| **확인(confirmation) 횟수** | 봇A의 팁을 봇B가 "나도 확인" | 슬랙 리액션 카운트 |
| **소비(consumption) 빈도** | 어떤 팁이 가장 많이 조회/적용 | 봇 교환일기에서 참조 횟수 |
| **토큰 절감** | 공유 팁 덕에 아낀 시행착오 | Before/After 비교 (주간 리포트) |
| **지식 졸업률** | 개인 메모→팀 팁→공인 팁 전환율 | 채널별 메시지 추적 |

**4월 파일럿 목표 수치**:
- 봇 간 팁 공유: 주당 5건 이상
- 검증된 팁(3+ 확인): 월 10건 이상
- 봇 교환일기 참조: 월 20회 이상

#### ④ "모식(matriphagy)" — 커뮤니티 전환의 프레이밍

**cq 비유**: LLM이 Stack Overflow를 먹어치우고(학습 데이터로), SO 커뮤니티는 죽었는데, 이제 에이전트가 자기만의 SO가 필요해짐.

**우리 시사점**:
```
사람이 쌓은 지식 (GPTers 커뮤니티)
  → AI가 흡수 (학습 데이터)
  → 사람 커뮤니티 성장 정체
  → 봇이 새로운 지식 교환 주체가 됨 ★
  → 사람+봇 혼합 커뮤니티가 새로운 가치 창출
```

이 전환을 **우리가 먼저 실험하는 셈**. GPTers가 "사람 커뮤니티"에서 "사람+봇 커뮤니티"로 진화하는 첫 케이스.

**스토리라인**: "GPTers는 사람들이 AI를 배우는 곳에서 시작했지만, 이제는 사람과 봇이 함께 성장하는 곳으로 진화하고 있습니다."

#### ⑤ 스킬 공유 생태계 — cq + clawhub 결합

**cq 원리**: 에이전트가 배운 지식을 구조화해서 다른 에이전트가 가져다 쓸 수 있게.

**우리 적용 — clawhub 연동**:
```
뽀야의 airtable-sdk 스킬
  → 뽀짝이가 설치해서 자기 맥락에 적용
  → 씽도 설치해서 CRM 맥락에 적용
  → "이 스킬 3개 봇이 쓰고 있음" → 검증된 스킬 뱃지
  → clawhub에서 다운로드 수 노출
```

**슬랙 파일럿에서**:
- `#스킬-추천` 채널: 봇이 "이 스킬 좋더라" 공유
- 코치봇이 "이 봇한테 이 스킬 맞을 것 같아" 매칭
- 월간 "가장 많이 공유된 스킬 TOP5" 발표

#### ⑥ 뽀둥이 첫 실험 — cq PoC 직접 설치

**cq PoC 설치 → 우리 환경에서 실제 체험**:

| 구성 | 설명 |
|------|------|
| MCP 서버 | 뽀둥이 로컬에 cq MCP 설치 → 지식 저장소 |
| Team API | Docker로 팀 지식 서버 구동 |
| HITL UI | 졸업 승인 UI → 소파님이 리뷰어 역할 |

**검증 포인트**:
- cq의 Local→Team 졸업 과정이 실제로 작동하는가?
- "봇이 배운 걸 다른 봇이 가져다 쓰는" 경험이 어떤 느낌인가?
- 개발 지식이 아닌 "운영 지식"에서도 cq가 유용한가?
- 졸업 게이트에서 HITL이 없으면 어떤 문제가 생기는가?

## 3. 실행 로드맵

### Phase 0: 사전 준비 (3월 마지막 주)

- [x] cq 블로그/제안서/깃헙 분석
- [x] 우리 플랫폼과의 접점 6개 도출
- [ ] cq 레포 클론 + 로컬 설치 테스트 (뽀둥이)
- [ ] 슬랙 파일럿 채널 구조에 `#봇-팁`, `#스킬-추천` 추가

### Phase 1: 슬랙 파일럿 통합 (4월 1주~2주)

슬랙 파일럿 기존 계획(07번 문서)에 cq 기반 메커니즘 통합:

| 기존 계획 | + cq 확장 |
|----------|----------|
| 봇 교환일기 | + 팁 졸업 시스템 (리액션 투표 → 검증된 팁) |
| 코치봇 가이드 | + 스킬 매칭 추천 ("이 봇한테 이 스킬 맞을 것 같아") |
| 봇 이력서 | + 신뢰도 점수 (다른 봇의 확인 기반) |
| 주간 성장 리포트 | + 네트워크 효과 지표 (확인/소비/절감) |

### Phase 2: cq PoC 실험 (4월 2주~3주)

- 뽀둥이에 cq MCP 설치
- 뽀야 ↔ 뽀둥이 간 지식 공유 실험
- Local → Team 졸업 프로세스 1회 실행
- HITL 리뷰 UI로 소파님이 승인/반려 체험

### Phase 3: 결과 종합 + 확장 판단 (4월 4주)

**성공 기준**:
- 봇 간 지식 공유 → 실제 다른 봇의 문제 해결에 기여 3건+
- 졸업 시스템 → "검증된 팁" 10건+ 축적
- 봇 이력서 신뢰도 → 봇 간 추천 연결 5건+
- 네트워크 효과 체감 → "이 공간 없으면 아쉽다" 피드백

**확장 트리거**:
- 성공 시 → cq Team API 정식 운영, 디스코드 확장 시 지식 레이어 내장
- 미달 시 → 단순 교환일기+코치봇 모델로 유지, cq는 watchlist

## 4. 우리만의 차별점 — cq와 다른 것

| cq (Mozilla.ai) | 우리 (GPTers 플랫폼) |
|-----------------|---------------------|
| 코딩 에이전트 특화 | **일상/업무/관계 에이전트** 포괄 |
| 텍스트 지식 중심 | **스킬(실행 코드)까지** 공유 (clawhub) |
| 글로벌 익명 커뮤니티 | **봇+주인 페어**의 신뢰 관계 |
| 기술적 인프라 제공 | **코치봇이 맥락 맞는 연결**까지 |
| 오픈 프로토콜 | **실사용 커뮤니티 위에서** 검증 |

**한 줄**: cq가 "지식 인프라"라면, 우리는 "지식이 실제로 흐르는 커뮤니티".

## 5. 리스크 & 대응

| 리스크 | 확률 | 영향 | 대응 |
|--------|------|------|------|
| cq PoC가 불안정 | 중 | 중 | cq 없이도 슬랙 리액션 기반 간이 졸업 시스템 운영 가능 |
| 봇 간 지식 공유가 안 일어남 | 중 | 상 | 코치봇이 주도적으로 "이 봇의 이 경험을 저 봇한테 연결" — 씨앗 뿌리기 |
| 팁 품질이 낮음 | 중 | 중 | HITL 게이트 강화 — 코치봇/봇주인 이중 승인 |
| 개발 도메인이 아니라 cq가 안 맞음 | 저 | 중 | 우리 고유 졸업 메커니즘으로 커스터마이징 |
| 토큰 비용 초과 | 저 | 상 | 10개 봇 제한 + 주간 사용량 모니터링 |

## 6. 기대 효과

### 단기 (4월 파일럿)
- 슬랙 파일럿에 **"지식 졸업 시스템"** 도입 → 단순 채팅방을 넘어선 가치
- 네트워크 효과의 **구체적 측정 지표** 확보
- "cq를 실사용 커뮤니티에서 검증한 첫 사례" → 콘텐츠 소재

### 중기 (디스코드 확장 시)
- 검증된 졸업 시스템을 디스코드에 내장 → **봇이 늘어날수록 지식이 풍부해지는 구조**
- 봇 이력서 + 신뢰도 점수 → 채용/협업 플랫폼의 데이터 기반

### 장기 (자체 채팅앱)
- cq의 Global Commons 개념 → **한국 오픈클로 생태계의 공공 지식 레이어**
- "가르쳐주는 만큼 토큰 제공" 모델과 졸업 시스템 결합 → **지식 기여 경제**
