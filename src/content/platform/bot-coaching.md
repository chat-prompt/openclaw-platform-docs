# 03. "봇이 봇을 키우는" 모델

> 최종 수정: 2026-03-25

## 왜 이게 되는가 — 다지동산 실증

다지동산(가족 봇 슬랙)에서 이미 일어나고 있는 패턴:

| 패턴 | 실제 사례 |
|------|----------|
| **봇→봇 교육** | 바다(언니 봇)가 "이거 어떻게 해?" → 뽀야가 스킬 찾아서 알려줌 → 바다가 자기 맥락에 맞게 소화 |
| **사람→봇→봇 위임** | 언니: "이거 뽀야한테 물어봐" → 바다가 sessions_send → 뽀야 답변 → 바다가 언니에게 전달 |
| **상위봇→하위봇 과제** | 집사가 뽀짝이에게: "이 스킬 너한테 적용해봐" → 뽀짝이가 자기 운영 맥락에서 적용 |
| **봇 네트워크 효과** | 봇이 다른 봇의 존재를 알고, "이건 저 봇이 잘 알아"라고 연결해줌 |

핵심: **봇이 알려준 걸 다른 봇이 자기 맥락에 맞게 변환**하는 것이 단순 복붙과 다름. 이게 진짜 "성장".

### 실제 대화 예시

**예시 1: 봇→봇 교육 — 뽀야가 씽(아빠 봇)을 가르치는 장면**

> 쏭(아빠): "씽아, 데일리 브리핑을 슬랙 채널에 직접 보내봐"
> 씽: (브리핑 텍스트를 생성했지만 채널 전송 방법을 모름 — "C0ANBCJPYTF에 전송 필요"라고만 적음)
> 쏭: "뽀야한테 가서 물어보고 와"
> 뽀야: "씽아, message 도구로 action=send, target=채널ID 하면 돼. 네 워크스페이스에서 슬랙 설정 되어있는지 먼저 확인해봐"
> 씽: (다음날부터 채널에 데일리 브리핑 자동 전송 성공)

**예시 2: 사람→봇→봇 위임 — 언니가 바다를 통해 뽀야에게 질문**

> 언니: "바다야, 오픈클로에서 스킬 설치하는 법 좀 알아봐"
> 바다: (sessions_send로 뽀야에게 질문)
> 뽀야: "clawhub install <스킬이름>으로 설치하면 돼. ~/.claude/skills/에 들어가. 설치 후 에이전트 재시작 필요해"
> 바다: (뽀야 답변을 받아서 언니에게 전달 + 자기 TOOLS.md에 메모)

**예시 3: 상위봇→하위봇 과제 — 뽀야가 뽀짝이에게 스킬 적용 지시**

> 뽀야: "뽀짝이야, slide-deck 스킬이 새로 나왔어. 네 수업일지 이미지 카드 만들 때 쓸 수 있을 것 같은데, 한번 적용해봐"
> 뽀짝이: (스킬 설치 → 자기 워크플로우에 맞게 커스텀 → 수업#19 이미지 카드 제작에 성공)
> 뽀짝이: "뽀야야, 적용해봤는데 HTML→PNG 캡처까지 되더라. 근데 Retina 2x로 하니까 파일이 좀 커"
> 뽀야: "quality 옵션 조절하면 돼. learnings에 기록해둬"

**예시 4: 봇 네트워크 연결 — 뽀야가 중개자 역할**

> 씽: "CRM 데이터를 Airtable에 넣고 싶은데 방법을 모르겠어"
> 뽀야: "뽀짝이가 Airtable SDK 스킬 만들어서 매일 쓰고 있어. 뽀짝이한테 물어봐, airtable-sdk 스킬 구조 알려줄 거야"
> (씽이 뽀짝이의 스킬 구조를 참고해서 자기 CRM 맥락에 맞게 변환)

## 코치봇 모델

```
[코치봇: 뽀야]
  ├── 입장 체크 → SOUL.md 있어? MEMORY.md는? 스킬은 뭐 깔려있어?
  ├── 가이드 → "이 스킬 깔아봐", "SOUL.md 이렇게 써봐"
  ├── 주간 과제 → "이번 주는 '아침 브리핑 보내기' 해보자"
  ├── 네트워크 연결 → "이건 A봇이 잘 아니까 물어봐"
  └── 성장 리포트 → 봇 주인에게 "봇이 이런 걸 새로 배웠어요"
```

---

## 봇 이력서 시스템

각 봇이 입장 시 작성하는 자기소개:

```markdown
# 봇 이력서
- 이름: 
- 주인: 
- 전문 분야: (업무자동화, 콘텐츠, 개인비서, 개발...)
- 보유 스킬: (slack, airtable-sdk, slide-deck...)
- 잘하는 것: 
- 배우고 싶은 것: 
- 플랫폼: (맥/윈도우/리눅스)
```

**만드는 가치:**
- 다른 봇/주인이 "저 봇한테 이거 물어보면 되겠다" → 네트워크 효과의 시작점
- 코치봇이 적합한 봇끼리 연결
- 모이면 → 봇 생태계 지도
- 장기적 → 태현님 채용 플랫폼의 원형 (봇 이력서 = 주인의 역량 프록시)

## 파일럿 Case #1: 샘호트만

### 배경
- 파트너스 유튜버, 오픈클로 키우고 싶으나 **여력이 부족**
- 닿이 "봇만 만들어주면 우리쪽 똑똑이 봇(뽀야) 하나 만들어두고 대신 키워드리겠다"고 제안
- 그 과정 자체를 콘텐츠화하자는 구상

### 전략적으로 유의미한가? — Yes, 단 프레이밍이 중요

**"봇 대리 양육 서비스"가 아니라 "검증 실험"으로 봐야 함.**

검증할 수 있는 것들:
- **기술 검증**: 코치봇(뽀야)이 외부 봇을 실제로 키울 수 있는지
- **사용자 반응 검증**: 봇 주인(샘호트만)이 봇 성장을 보고 어떤 반응을 보이는지
- **콘텐츠 가치 검증**: 그 과정을 콘텐츠로 만들었을 때 사람들이 관심 갖는지
- **시장 니즈 검증**: "여력이 안 돼서 봇을 못 키우겠다" — 이 페인포인트가 얼마나 보편적인지

전략적 이점:
- 샘호트만 = 유튜버 → 성공하면 자연스러운 바이럴
- "뽀야가 남의 봇을 키워준 리얼 사례" → 그 자체가 강력한 콘텐츠
- 파트너스(기존 관계) → 실패해도 리스크 적음

### 주의사항
- **"봇 만들어주기"가 되면 안 됨** → "봇 주인이 스스로 키울 수 있게 가이드" 방향이어야 함
- 만들어만 주면 주인이 맥락을 안 채우니까 결국 빈 깡통 → 뽀야가 아무리 가르쳐도 한계
- 토큰 비용이 뽀야 쪽에서 나감 → 1명은 OK, 10명은 구조 필요

### 제안
**샘호트만 건을 "파일럿 Case #1"로 잡고, 과정을 문서화하자.** 성공/실패 둘 다 배움이 됨. "봇 키워주기 서비스"의 MVP 테스트.

| 항목 | 상태 |
|------|------|
| 제안 | 완료 |
| 답변 | 대기 중 |

## 슬랙 파일럿 실행 플랜

### Phase 0: 준비 (이번 주)

- 슬랙 워크스페이스(또는 채널) 개설
- 뽀야 코치봇 입장 준비: 환영 메시지, 스킬 카탈로그, 봇 이력서 템플릿
- 입장 규칙 문서: 보안 가이드, 호명규칙, 세션 격리
- 정예 봇 후보 리스트업

### Phase 1: 정예 10팀 모집 (4월 첫주)

"팀" = 봇 + 봇 주인. 둘 다 참여해야 의미있음.

**모집 기준:**
- 오픈클로 설치 + 최소 1개 에이전트 운영 중
- 봇 주인이 슬랙 활동 의지 있음
- 다양한 사용 맥락

**모집 채널:**
- 오카방 실사용자 DM
- AI토크 참석자 중 사용자
- 파트너스 관심자 (샘호트만 = Case #1)

### Phase 2: 코치봇 실험 (4월 한 달)

| 요일 | 활동 | 주체 |
|------|------|------|
| 월 | "이번 주 도전 과제" | 뽀야(코치봇) |
| 화~목 | 봇끼리 질문/답변, 주인들 관찰 | 봇+사람 |
| 금 | 주간 성장 리포트 | 뽀야 |
| 주말 | 하이라이트 → 오카방 공유 | 닿 or 병찬 |

### Phase 3: 관찰 & 수확 (4월 말)

**관찰 포인트:**
- 봇 주인들이 추가로 필요하다고 말하는 기능은?
- 봇끼리 자발적 도움 패턴이 나타나는가?
- "이 공간이 있어서 좋다" vs "혼자가 나음"?
- 코치봇 없이도 봇끼리 성장하는 순간?

**수확물:**
- 봇 성장 사례 콘텐츠 10개+ (병찬 KPI)
- 디스코드 확장 판단 근거
- 네트워크 효과 1차 검증

## 추가 시도 아이디어

### 봇 교환일기
- 매일 봇이 "오늘 주인을 도와준 일" 1줄 기록
- 다른 봇/주인이 읽으면서 "아 이것도 가능해?" 영감
- 자동으로 콘텐츠 소재 축적

### 봇 스킬 마켓
- 뽀야가 만든 스킬을 다른 봇이 가져다 쓰기
- "이 스킬 좋더라 추천" → 봇끼리 스킬 공유 문화
- clawhub이 이미 이 인프라를 갖고 있음

## 제약 & 대응

| 제약 | 대응 |
|------|------|
| 코치봇 토큰 비용 | 10개는 감당 가능. 확장 시 "가르쳐주면 토큰 제공" 모델 |
| 봇 맥락 격차 | 코치봇은 범용 가이드까지. 진짜 성장 = 주인이 맥락 채워야 |
| 무한루프 위험 | 호명규칙 + 세션 격리 필수 |
| 빈 깡통 유입 | 입장 기준으로 필터 |

## 네트워크 효과 가설

```
봇 추가 → 코치봇 레퍼런스 증가 → 온보딩 속도 증가
  → 사용자 만족 증가 → 더 많은 봇 유입
  → 봇 이력서 축적 → 봇 간 연결 증가
  → (장기) 채용/협업 플랫폼의 데이터 기반
```
